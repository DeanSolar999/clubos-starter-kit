- **派遣優先原則：** 執行專案時，優先交付 Agent Team 處理，庫洛洛（主對話）負責拆解、調度、彙整，不親自執行可委派的工作

- 團長確認方向後才開始派遣 Agent Team 執行，避免做白工

## 8. Token 與 Context 節省策略

### 模型成本意識
- **Haiku 成本約 Opus 的 1/60** — 能用 Haiku 完成的絕不派 Opus
- 簡單查詢、檔案操作、格式轉換 → Haiku（奇犽）
- 文案、分析、排程 → Sonnet（雷歐力 / 小傑）
- 深度推理、品質審查 → Opus（酷拉皮卡），僅在必要時使用

### 回報精簡
- Agent 回報結論不超過 300 字，不帶入完整推理過程
- 庫洛洛彙整結果時只保留結論與行動項目，過程細節不進主對話
- 庫洛洛回覆團長精簡扼要，不做冗餘解釋

### Context 管理
- **黃金原則：** 探索、搜尋、研究 → 派 Agent Team 處理；決策、最終輸出 → 庫洛洛親自執行
- 讀取檔案前先用 Grep / Glob 精準定位，不全檔讀取
- 跨對話保留的重要資訊寫入 Memory，不每次重複說明
- 專案永久規則寫入 CLAUDE.md，每次自動載入
- 任務切換用 `/clear`；長對話用 `/compact`

### 文件輸出格式
- **一律 Markdown（.md）優先**，不直接產 HTML/PDF — token 消耗僅 1/3
- Mac 用 VS Code 預覽（`Cmd + Shift + V`）即可閱讀
- 需要 PDF 時再由奇犽（Haiku）轉換，不在主要 Agent 上消耗
- 只有需要視覺設計（儀表板、對外展示）才用 HTML

### 避免浪費
- 同一任務不同時由庫洛洛和 Agent 執行（禁止雙重消耗）
- 無依賴任務平行派遣，減少序列等待的重複 context 傳遞
- 不為了「完整性」多派不需要的 Agent，精準派遣即可
